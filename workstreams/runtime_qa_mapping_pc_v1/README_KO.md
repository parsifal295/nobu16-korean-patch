# PC 런타임 QA 진입·재현 지도 v1

## 결론

이 문서는 현재 Wave7 PC 후보를 읽기 전용으로 대조해, 실제 게임에서
확인해야 하는 항목을 **확정된 장면 식별**, **재현 계획**, **아직 없는
근거**로 분리한 것이다. Switch 자료는 열지 않았고, 게임 실행·클릭,
Steam 쓰기, Git 변경도 하지 않았다.

정적으로 확정된 것은 텍스트 ID/현재 문구/런타임 토큰/인접 이벤트군이다.
반면 Base 역사 이벤트 6건과 PK 인물 대사 `2:330`, `2:551`, `2:628`의
실제 발동 조건·호출 지점은 PC 파일만으로 복원되지 않았다. 아래의
`재현 계획`은 그럴듯한 추측을 확정 사실처럼 쓰지 않기 위해, 실제 QA에
필요한 세이브·상태를 명시한 것이다.

## 읽은 근거와 범위

- 수집기: [collect_runtime_qa_mapping_pc_v1.py](collect_runtime_qa_mapping_pc_v1.py)
- 기계 판독 결과: `runtime_qa_mapping_pc_v1.evidence.json`
  - SHA-256: `1D8F07C03A823A3BEE9B7141C6470CC4AFAA33150D49776EC1E95BAB4103BD20`
- 입력: PC Base JP 원본, Wave7 현재 KO 후보, PC EN/SC/TC 참조본 및 기존
  PC-only 감사 자료. Switch 경로는 제외했다.
- 현재 대상 해시:
  - Base `MSG/JP/ev_strdata.bin`: `25D9C029…0D5834`
  - Base `MSG/JP/msggame.bin`: `83C4DF93…3C279E`
  - PK `MSG_PK/JP/msgev.bin`: `134F6356…4CF154`
  - PK `MSG_PK/JP/msggame.bin`: `31950B82…A34BDB`
- Base 역사 이벤트 6개 ID를 `FLOW/*.bsf`의 직접 little-endian 32-bit 값으로
  검색했으나 모두 0건이다. `FLOW_PK`/big-endian의 고립된 바이트 일치는
  호출 관계를 증명하지 못하므로 발동 조건 근거로 쓰지 않는다.
- 각 이벤트의 정확한 원문 앵커도 `FLOW`·`SCENARIO`·`PARAM` 등 PC 데이터에
  중복 저장돼 있지 않았다. 따라서 텍스트 테이블은 장면 식별에는 충분하지만
  조건 역추적에는 충분하지 않다.

## `msggame` — 현재 런타임 대사

### 친선 실패 6건 (Wave7 적용분)

현재 출력은 Base `6:1518–1520`, PK `6:1524–1526` 모두 `025032` 런타임
세력명 뒤에 `측과의 친선은`을 두고, 고정 한국어 조사로 끝낸다. 토큰 자체는
바이트 보존되어 있으므로 실제 세력명이 긴 경우의 줄폭은 아직 화면에서만
판정할 수 있다.

| Base / PK | 발생 조건으로 식별된 실패 사유 | 재현 계획 | 필수 캡처 |
|---|---|---|---|
| `6:1518` / `6:1524` | 거리 부족 | 외교의 친선 실행에서 멀리 떨어진 세력을 선택해 실패 상태를 만든다. | 세력명 + `거리가 너무 멀어 계속할 수 없습니다.` 2줄. |
| `6:1519` / `6:1525` | 비용 부족 | 같은 친선 실행에서 금전이 부족한 상태를 만든다. | 세력명 + `비용이 부족해 계속할 수 없습니다.` 2줄. |
| `6:1520` / `6:1526` | 중개자 부재 | 대상과의 친선에 필요한 중개자가 없는 상태를 만든다. | 세력명 + `중개자가 없어 계속할 수 없습니다.`의 수동 3분절. |

각 사유는 Wave7 감사의 PC JP 원문과 `025032` 스키마로 확인됐다. 실제 QA는
가능하면 짧은 세력명 1개와 긴 세력명 1개를 모두 써야 한다. 현 문구는
세력명 종성에 따른 조사를 쓰지 않으므로, 종성 검사는 문법보다 폭·토큰
삽입 검증 목적이다.

## 실제 QA 기록 형식

각 캡처에 아래를 함께 남긴다.

1. 리소스/ID, 사용한 세이브·진입 순서, 선택한 세력·대상·시설명.
2. 선택 해상도, 해상도 변경 여부, 변경했다면 게임을 완전히 종료·재실행했는지.
3. 화면 전체 캡처와 판정: 시각 줄 수, 잘림/자동 4줄, 토큰 누락, 조사·공백,
   색상/특수문자.

해상도를 바꿨다면 프로세스를 완전히 종료 후 재실행한 화면만 이미지 적용
증거로 취급한다.
