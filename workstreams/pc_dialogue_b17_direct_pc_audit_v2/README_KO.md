# PC MSGGAME B17 직접 대조 품질 감사 v2

이 문서는 Steam PC의 현재 한국어 `msggame.bin`과 Steam PC 원본 일본어 `msggame.bin`만 직접 대조한 B17 감사 기록이다. 기존 B17 후보·기존 감사·다른 언어판은 근거로 읽지 않았다.

- 좌표 표기는 `B17:<record>:<literal>`이다.
- literal은 검증된 `07 07 01 ... 07 07 02` UTF-16LE 구간만 뜻한다.
- 이 문서의 `정정 방향`은 후보 파일이나 Steam 반영이 아니다. 런타임 토큰·문맥·화면 폭 확인이 필요한 것은 별도로 HOLD로 분리했다.
- B17의 좌표 전수 대조는 블록 단위의 구조·문자열 대조일 뿐, 게임 전체의 문학적 완성이나 실게임 렌더링 완료를 뜻하지 않는다.

## 전수 범위와 구조 검증

| 리소스 | B17 record (KO / JP) | B17 block bytes (KO / JP) | literal slot (KO / JP) | 표시 가능한 literal (KO / JP) | LF literal / LF 수 (KO / JP) | 다중 literal record |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Base | 33 / 33 | 2,502 / 2,198 | 66 / 66 | 64 / 64 | 25 / 25 | 15 |
| PK | 1,159 / 1,159 | 88,617 / 77,361 | 2,256 / 2,256 | 2,238 / 2,238 | 814 / 824 | 652 |
| 합계 | 1,192 / 1,192 | — | 2,322 / 2,322 | 2,302 / 2,302 | 839 / 849 | 667 |

검증 결과:

- Base 66개, PK 2,256개, 합계 2,322개 literal 좌표가 JP와 일대일로 정렬된다. literal 수가 다른 record는 0개다.
- 각 record에서 literal 본문만 제거해 만든 opaque skeleton도 Base 33/33, PK 1,159/1,159가 JP와 일치한다. 즉 이 감사에서 발견한 정적 수정은 proven literal 밖의 제어 바이트를 건드릴 이유가 없다.
- Base/PK B17 한국어 literal에서 U+FFFD, 히라가나, 가타카나, CJK 한자, ASCII `...` 연속점은 0건이다. 단 PK `B17:282:0`의 `Color.Blue`/`Color.Default` 텍스트는 JP에도 동일하게 있어 별도 런타임 HOLD다.
- LF의 위치·개수는 KO/JP가 동일하다. 이 수치만으로 실제 대화창의 줄 수나 화면 폭 적합성을 증명할 수 없으므로 자동 개행 삭제는 하지 않는다.

### 안전한 표기 정규화

진행 목표의 괄호 짝이 ASCII 여는 괄호 `(`와 전각 닫는 괄호 `）`로 섞여 있다. 문장 의미나 토큰은 바꾸지 않고 한 쌍으로만 정규화할 수 있다.

| record | 현재 조합 |
| --- | --- |
| PK `B17:296`, `300`, `304`, `463`, `467`, `471`, `696`, `790`, `1093`, `1157` | 각 record의 첫 literal 끝 `(` + 다음 literal `/N）` |

프로젝트 UI 표기 기준에 따라 `(`/`)` 또는 `（`/`）` 중 한 쌍으로 맞추되, 두 literal과 중간 제어 바이트는 그대로 유지한다.

## HOLD — 런타임·문맥 확인 전에는 정적 치환하지 않을 항목

| 좌표 | 관찰 | HOLD 이유 |
| --- | --- | --- |
| PK `B17:226:0-3` | 이름 삽입 제어 바이트 사이에 `은(는)`이 literal로 존재한다. | 이름 끝소리에 따라 조사가 달라진다. 단순히 `은` 또는 `는` 하나로 고르면 다른 이름에서 문법이 깨질 수 있다. 실제 출력과 token 처리 방식을 확인해야 한다. |
| PK `B17:920:0-1` | 첫 literal은 이미 `… 나를 따르라!`, 이름 token 뒤 literal은 `을(를) 따르라!`다. | 중복 명령과 조사 표기가 모두 문제지만 token 뒤 피삽입 이름의 받침을 알 수 없다. 런타임 문장 재구성·실게임 확인 후 처리한다. |
| PK `B17:510:0-2` | 이름 token 앞뒤 조합이 `동군` + token + `에게 배반해`다. | 현재는 조사의 대상과 어순이 어색하지만, token이 가문·무장·부대 중 무엇을 출력하는지 이진 literal만으로 확정하지 않는다. 실출력 확인 후 문장 전체를 조정한다. |
| PK `B17:282:0` | `Color.Blue` / `Color.Default`가 JP와 KO 모두 literal에 있다. | 번역 잔재가 아니라 런타임 색상 markup일 수 있다. 문자열에서 제거·번역하면 색상 처리 또는 parser를 손상시킬 수 있다. |
| Base `B17:5:3`; PK `B17:226`, `510`, `920`, `950`, `971`, `991` 등 | 다수 literal이 이름·부대 token과 결합된다. | 위의 명확한 정적 어휘·문법 문제 외에는 token 전후 조사를 추측으로 바꾸지 않는다. |
| B17 전체 LF 849개 | KO/JP LF 좌표·개수가 동일하다. | 수동 LF가 문맥 전환인지 UI 폭 대응인지는 실게임 대화창에서만 판단 가능하다. 일괄 삭제나 무근거 재배치는 하지 않는다. |

## 작업 경계와 결론

- 새 번역 후보, 게임 파일, Steam 설치본, Git 인덱스·커밋·푸시·릴리스는 변경하지 않았다.
- 이 보고서는 B17의 현행 PC KO와 pristine PC JP만 근거로 했다.
- 후속 후보를 만들 때는 위 고신뢰 좌표만 먼저 opaque record 단위로 묶고, HOLD는 별도 런타임 검증 대상으로 남긴다.
- B17 2,322 literal의 구조·대조 범위는 완료했지만, 이것이 전체 게임 텍스트의 전면 품질 완료나 실게임 줄바꿈 완료를 뜻하지는 않는다.
