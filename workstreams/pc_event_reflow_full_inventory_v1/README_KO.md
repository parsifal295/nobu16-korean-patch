# Steam PC 이벤트 수동 개행 전수 구조 감사 v1

## 결론

현재 Steam PC 이벤트 테이블을 전수 구조 스캔한 결과, 실제로 색상 태그를 가로질러
이름·지명 등이 끊기는 유형은 **PK `msgev.bin`의 46개 레코드 / LF 47개**뿐이다.
이 46개는 모두 현재 private `W58` 후보에 반영되어 있고, 후보에서는 태그 내부 LF가
0개이며 각 행이 2~3줄·줄당 912 px 이하임을 다시 확인했다.

반대로 `줄바꿈을 전부 없애는` 방식은 사용하지 않는다. 태그 바깥의 기존 수동 개행은
문맥상 의도된 행 구분일 수 있으므로 그대로 보존한다. 남은 PK 폭 검토 행 99개도
문맥·어절 위치를 사람이 확인하기 전에는 자동 후보로 만들지 않는 `WIDTH HOLD`다.

## 범위와 입력

Steam Base에는 요청명과 같은 `MSG/JP/msgev.bin` 파일이 없다. Base 이벤트 테이블의
실제 경로는 `MSG/JP/ev_strdata.bin`이다. 따라서 아래 두 Steam PC 이벤트 테이블을
직접 읽었다.

Switch·다른 언어 테이블·네트워크는 읽지 않았다. 일본어 원문은 이 구조 판정에 필요하지

쓰기 작업을 하지 않았다.

## 검사 방법과 보수적 기준

- 두 테이블은 LZ4 해제 → 메시지 테이블 parse/rebuild → 재압축 round-trip을 모두 통과했다.
- 모든 개행은 LF이며 CR/CRLF는 없다. Base는 LF 8,160개(5,006행), PK는 LF 11,540개(6,681행)다.
- 색상 span은 `ESC C A/B/C`로 열고 `ESC C Z`로 닫는 구조로 읽었다. 두 테이블 모두
  열린 span/닫힘 span의 불일치와 태그 외 제어문자는 없었다.
- PK의 확정 렌더링 폭 기준은 현재 이벤트 글꼴의 912 px, 최대 3줄이다.
- `unwrapped` 하한은 태그와 LF를 화면 폭 0으로 제외한 전체 가시 폭을 `ceil(폭 / 912)`로
  계산했다. 4 이상이면 **어떤 행 배치만으로도** 3줄에 들어갈 수 없다는 하한이다.
  반대로 3 이하는 어절 위치·런타임 치환 때문에 실제 배치가 안전하다는 뜻은 아니다.

## 전수 요약

| 항목 | Base `ev_strdata` | PK `msgev` |
|---|---:|---:|
| 색상 span 내부 LF 레코드 / LF | 0 / 0 | 46 / 47 |
| 비어 있지 않은 표시 행이 3줄 초과 | 0 | 0 |
| 끝 LF만 가진 행 | 3411 (3개 내용 행 + 끝 LF) | 0 |
| 912 px 기준 unwrapped 하한 4줄 이상 | 868 | 0 |
| 현재 수동 행 중 912 px 초과 | 3,335 | 108 |

Base의 912 px 수치는 PK의 렌더러 폭을 넉넉하게 적용한 **분류용 하한**일 뿐, Base 이벤트
창의 실제 폭 계약을 뜻하지 않는다. 따라서 Base의 넓은 행은 이 보고서에서 자동 수정 대상이
아니며, 별도 Base 렌더러 QA 전까지 `BASE WIDTH HOLD`다.

## Base 분리 보류

Base에는 색상 span 내부 LF가 없어서 PK와 같은 이름·지명 분절 결함은 검출되지 않았다.
다만 912 px를 공통 하한으로 가정했을 때 unwrapped 최소 4줄인 행은 868개다.

- 686개: 런타임/printf 없는 정적 문자열이지만 Base 실제 렌더러 폭 계약이 없으므로
  `BASE WIDTH HOLD`다.
- 182개: 런타임 또는 printf가 있어 `BASE RUNTIME/CONTROL HOLD`다. 아래 목록이며,
  868개 전체 목록에서 이 목록을 제외한 것이 정적 686개다.
- ID 3411은 세 개의 비어 있지 않은 내용 행 뒤에 끝 LF만 가진 유일한 행이다. 네 번째
  표시 행으로 단정할 수 없으므로 그대로 보류한다.
