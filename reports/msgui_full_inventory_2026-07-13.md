# `msgui` 전체 UI 인벤토리 보고서 (2026-07-13)

> **개발 전용 / 공개 패치 포함 금지**
> 이 보고서와 `workstreams/msgui_full/catalog_v2`, `workstreams/msgui_full/inventory`는 분석·번역·QA용이다. 완전한 상용 원문이 들어 있는 카탈로그와 원본 추출물은 배포하지 않는다. 공개 배포물은 검증된 한국어 변경분과 이를 재구성하는 소형 recipe만 포함해야 한다.

## 결론

`MSG_PK/*/msgui.bin`의 EN/JP/SC/TC 네 언어를 모두 대조한 결과, UI 문자열은 언어마다 **ID 0~5099의 연속된 5,100행**으로 정확히 정렬된다. 교차 언어의 유일한 안정 키는 숫자 ID이며, 문자열 바이트 오프셋은 언어마다 달라 재사용할 수 없다. 따라서 한국어 패치는 **SC를 파일 기반 베이스로 삼되 ID별 문자열을 교체하고 오프셋 테이블을 다시 계산**하는 방식이 안전하다.

이번 작업은 기존 개발용 추출물만 읽었다. 게임 실행, 프로세스 메모리 패치, 레지스트리 변경, 설치본 쓰기는 전혀 수행하지 않았다.

## 입력과 무결성

개발용 원문 카탈로그는 `KR_PATCH_WORK/workstreams/msgui_full/catalog_v2/msgui.catalog.p3.jsonl`이며 SHA-256은 `80E1B46F829F810FA4E55C412D92ED8FC8B48CEFB11FC826C3CD81FED83232BA`다. 인벤토리 레코드는 원문을 복제하지 않고 이 카탈로그의 동일 ID를 참조한다.

| 언어 | 압축 원본 크기 | 압축 원본 SHA-256 | 추출 RAW 크기 | RAW SHA-256 |
|---|---:|---|---:|---|
| EN | 84,827 | `FB9C4698989319AE9A9BF138F6DC963DC106124FC60D19308D0BF1AD35BE3965` | 219,108 | `ECFDAF31F393EA5A442C3D416910A80CAD3EB9DE9ED5AD1BA6559D0619BEDBAD` |
| JP | 62,251 | `B099AD053DB48E3040E76D3FB7F6785C8E5A94D7C5D62EE02C137489001C2B35` | 99,864 | `E3B36C5A34825EB22EC21A562F10A622905076EB4219D5BFA7D56C995A80EEFE` |
| SC | 60,829 | `C2C69FDF09D9BE06E14F03C4F40562ADD0CA247EE0D50FC3E06EF501524B5E82` | 85,580 | `D0699841A1F96BBF08420F319A271A2F960F7851164082EA3170BC4CA3A32976` |
| TC | 60,814 | `000355A838A3805A7BEA3FB9D8F84C42B178593007D4B86DF93B2B81BCB2C513` | 85,824 | `1EFD35F9E4C8FD694829B39C770862D0062FCF9C0326538AD9CFA5DDCA426A3C` |

네 RAW 모두 ID가 연속·유일하고, 오프셋은 짝수이면서 엄격히 증가한다. 각 span은 UTF-16LE 문자열과 종단 NUL로 정확히 닫힌다. SC에만 논리 끝 뒤 2바이트의 0 패딩이 있고 나머지는 패딩이 없다. 기존 JSONL과 RAW 문자열도 ID별로 정확히 일치한다.

## 바이너리 구조와 안정 키

네 언어의 공통 구조는 다음과 같다.

| 항목 | 값 |
|---|---:|
| 블록 수 | 1 |
| 블록 시작 | `0x0C` |
| 테이블 상대 위치 | `0x14` |
| 테이블 절대 위치 | `0x20` |
| 오프셋 테이블 크기 | `0x4FB0` = 5,100 × 4바이트 |
| 문자열 풀 절대 위치 | `0x4FD0` |

반면 동일 ID가 같은 바이트 오프셋을 갖는 경우는 매우 적다.

| 언어 쌍 | 같은 오프셋의 ID 수 |
|---|---:|
| EN–JP | 1 |
| EN–SC | 2 |
| EN–TC | 2 |
| JP–SC | 1 |
| JP–TC | 1 |
| SC–TC | 9 |

네 언어 모두 같은 오프셋인 것은 ID 0 하나뿐이다. **다른 언어의 오프셋을 복사하는 패처는 금지**하고, 빌드 시 대상 문자열 길이로 테이블 전체를 재계산해야 한다.

## 빈 문자열과 공백 행

| 언어 | 빈 문자열 | 공백만 | 실문자 |
|---|---:|---:|---:|
| EN | 1,048 | 66 | 3,986 |
| JP | 1,038 | 4 | 4,058 |
| SC | 1,038 | 122 | 3,940 |
| TC | 1,038 | 122 | 3,940 |

JP/SC/TC의 빈 ID 집합은 완전히 같다. EN은 ID 460~469가 추가로 비어 있으므로 EN을 내용 기준의 기준표로 삼으면 이 10행을 놓칠 수 있다.

SC 공백 행 122개 중 네 언어 모두 공백인 것은 3개뿐이다. 나머지 **119개는 다른 언어에 실제 문구가 있어 `needs_context` 대상**이며 자동 제외하면 안 된다. 완전 빈 1,038행과 전 언어 공백 3행만 현재 안전한 `SKIP` 후보다.

| 언어 | 평균 | 중앙값 | P90 | P95 | 최대 |
|---|---:|---:|---:|---:|---:|
| EN | 18.4780 | 12 | 45 | 60 | 211 |
| JP | 6.7875 | 4 | 17 | 22 | 74 |
| SC | 5.3869 | 4 | 13 | 17 | 65 |

영어는 CJK 언어보다 훨씬 길어 화면 폭 판단의 참고가 되지만, 한국어 폭은 실제 Noto Sans KR 기반 래스터와 게임 화면에서 별도로 측정해야 한다. 코드포인트 수만으로 overflow 합격을 판정하지 않는다.

## 구조 분류와 토큰 위험

SC 기준 상호 배타적 자동 분류는 다음과 같다.

| 구조 분류 | 행 수 |
|---|---:|
| `all_empty` | 1,038 |
| `all_whitespace` | 3 |
| `sc_blank_other_text` | 119 |
| `short_ui_candidate` | 2,606 |
| `medium_prompt_candidate` | 599 |
| `long_help_candidate` | 25 |
| `multiline_static` | 58 |
| `dynamic_printf` | 626 |
| `styled_control` | 11 |
| `styled_dynamic` | 15 |

SC의 토큰 현황:

- printf: 641행, 987개
- ESC 스타일 토큰: 26행, 54개. 모두 `ESC + C + 1문자`의 3코드 단위이며 네 언어에서 ID와 순서가 같다.
- LF: 128행, 180개
- PUA: 20행, 23자, 16종
- NUL 및 Unicode replacement character: 없음

빌드 검증기는 printf와 ESC 토큰의 **개수뿐 아니라 순서**를 보존해야 한다. 인식되지 않은 `%`, NUL, 잘린 ESC 단위는 즉시 실패시켜야 한다. LF 수 변경은 사람 검수와 명시적 override 없이는 허용하지 않는다.

특히 ID 3904는 SC와 TC의 printf 순서가 서로 다르다. SC는 `%s,%s,%d`, TC는 `%s,%d,%s`이므로 한국어 빌드는 SC 순서를 보존하고 고위험 수동 QA 대상으로 둔다. 전체 위험도 자동 분류는 high 76행, medium 945행, low 4,079행이다.

## 중복과 번역 메모리

정확한 Unicode 문자열 일치 기준 SC 실문자 중복은 150그룹/333행이다. 그중 다른 언어 문맥이 갈리는 의미 충돌은 77그룹/180행이다. 따라서 **SC 문자열 하나만을 키로 한 자동 번역 전파는 금지**한다.

| 중복 조건 | 그룹 | 포함 행 | 절감 가능 중복 행 |
|---|---:|---:|---:|
| SC 정확 일치 | 150 | 333 | 183 |
| SC 일치지만 다국어 문맥 충돌 | 77 | 180 | 해당 없음 |
| EN/JP/SC/TC 튜플 정확 일치 | 81 | 169 | 88 |
| 다국어 튜플 정확 일치 + 토큰 없음 | 74 | 154 | 80 |

대소문자 무시 비교는 참고용으로만 제공한다. 이 경우 151그룹/335행이고 의미 충돌은 78그룹/182행이다. 번역 메모리는 EN/JP/SC/TC 네 문자열의 정확한 튜플 hash가 같은 경우에만 **제안**할 수 있으며, 자동 확정은 하지 않는다.

## 번역 우선순위

| 우선순위 | 행 수 | 용도 |
|---|---:|---|
| P0 | 9 | 실제 화면에서 확인된 메인 메뉴 seed: IDs 1, 2, 5, 56, 85, 87, 88, 97, 178 |
| P1 | 436 | 공통 탐색·상태·시스템 후보. 화면 근거를 붙여 확정 |
| P2 | 2,760 | 나머지 짧은 정적 UI 및 중간 길이 프롬프트 |
| P3 | 710 | printf, ESC, 여러 줄 등 구조 위험 문자열 |
| P4 | 144 | 긴 도움말 및 `sc_blank_other_text` |
| SKIP | 1,041 | 전 언어 빈 문자열 1,038 + 전 언어 공백 3 |

P1~P4는 기계적 제안이며 의미나 실제 노출 빈도를 보증하지 않는다. 각 행은 `priority_basis`, `priority_confidence`, `surface_evidence`를 별도로 보존하고 실제 게임 화면 근거가 생길 때 수동 우선순위를 확정해야 한다.

현재 P3 카탈로그 상태는 translated 279행, untranslated 3,783행, empty 1,038행이며 buildable은 279행이다. 한국어 glyph demand는 총 238자이고 이 중 한글 음절은 226자다. 폰트 생성기는 이 수요 파일을 입력으로 사용하되, 실제 게임 화면 QA 전에는 release 가능으로 표시하지 않는다.

## 생성 산출물

경로: `KR_PATCH_WORK/workstreams/msgui_full/inventory/`

- `records.jsonl`: 5,100행의 metadata-only sidecar. 원문 대신 카탈로그 ID·hash·길이·오프셋·토큰·분류를 기록
- `summary.json`: 구조, 길이, 공백, 토큰, 우선순위, 위험도 요약
- `duplicates.json`: 정확 중복, 다국어 튜플, 의미 충돌 그룹
- `token_catalog.json`: printf/ESC/LF/PUA 구조 목록
- `priority_buckets.json`: 우선순위별 ID 목록
- `glyph_demand.json`: 현재 한국어 번역의 폰트 수요
- `source_manifest.json`: 입력 경로·크기·SHA-256 및 비변경 선언
- `inventory_manifest.json`: 산출물 SHA-256
- `README_DEVELOPMENT_ONLY.md`: 개발·배포 경계

생성 도구는 `KR_PATCH_WORK/tools/build_msgui_full_inventory.py`다. 같은 P3 입력으로 두 번 연속 실행한 결과 9개 산출물의 SHA-256은 모두 동일했다(`ChangedCount=0`). `records.jsonl`은 정확히 5,100행이며 `text`, `source_*`, `translation` 원문 필드가 없음을 별도 검사했다.

## 배포 경계와 다음 QA

1. `catalog_v2`, 원본 RAW/JSONL, `inventory` 전체는 개발 전용으로 유지한다.
2. 공개 패치는 원본 전체 파일이나 전체 원문을 포함하지 않고, 지원되는 기준 SHA에 적용되는 검증된 한국어 변경 recipe와 공개 가능한 생성 도구·폰트 recipe만 포함한다.
3. 설치 전 기준 SHA 검증, 임시 경로 재구성, 결과 SHA 검증, 원자적 교체, 백업/복구를 적용한다.
4. 프로세스 메모리 패치, 런타임 후킹, 레지스트리 의존 방식은 설계 범위에서 제외한다.
5. P0부터 실제 화면을 확인하면서 한글 glyph, 토큰 치환, 줄바꿈, 잘림, 세로/가로 배치, 장수명 등 장문 폭을 QA한다.

자동 분류는 번역 큐를 만드는 수단이지 의미 판정이 아니다. 119개의 SC 공백/타언어 실문자 행, 중복 의미 충돌, ID 3904, 그리고 토큰·여러 줄 문자열은 반드시 문맥 및 게임 화면 검수를 거쳐야 한다.
